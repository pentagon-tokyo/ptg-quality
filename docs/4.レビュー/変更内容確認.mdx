# 変更内容確認

レビューでは、品質や工数をレビューしやすくなるような工夫が必要です。

* 変更内容を列挙する（ソースコードの差分内容と対応していること）
* チケットの期待値と変更内容に差分がある場合、差分がある理由について説明すること

変更内容記載例

> * users: ユーザーを個別に論理削除できるAPIを追加(*1)
> * users: 既存の親ユーザーに紐づく子ユーザーを新規作成できるAPIを追加
> * ユーザー追加のテストコード追加
> * ユーザー削除のテストコード追加
> 
> (*1) 親ユーザー単位ではなく、子ユーザー単位で論理削除する必要があったため、APIを追加しました。

## 修正範囲の連携

チケット以外の部分の修正を行う場合(リファクタリング観点・命名の変更・微修正)、レビューの妨げとならないよう配慮すること。

* PRの説明文にて補足する
* 事前にチャットやチケットなどで相談する（想定スケジュールを逸脱するような作業を独断で行わないこと）

## デザイナーのレビュー参加

UI変更がある場合、デザイナーのレビューを行うこととする。  
デザイナーのレビュー参加により、以下のメリットが享受できる。

* 実装着手後に仕様変更点があった場合などに連携漏れがあったとしても、レビュー時にキャッチアップできる
* エンジニアが見逃す可能性のあるデザイン観点でのレビューができる
* デザイナーが開発の進捗を把握できる
* 作業者がデザイナーとコミュニケーションとれる
