# 変更内容確認

レビューには実装者1名、レビュアー1名以上の2名以上が関わります。  

UI変更がある場合、レビュアーはデザイナー1名以上、エンジニア1名以上が参加します。
UI変更がない場合、レビュアーはエンジニアのみが参加します。

## 実装者が気をつけるべきこと

[プルリクエスト作成](/3.実装/プルリクエスト作成) を参考に、レビュースコープがレビュアーへ正しく伝わるよう配慮すること。

## デザイナーによるレビュー

デザイナーは仕様やデザイン観点でのレビューを行う。

* 仕様通りの挙動となっているかをエビデンスで確認
* デザイン観点の確認

また、デザイナーがレビュー参加することで以下の恩恵がある。

* 実装着手後に仕様変更点があった場合などは、最悪レビュー時にキャッチアップできる
* デザイナーが開発の進捗を把握できる
* 作業者がデザイナーとコミュニケーションとれる

## エンジニアによるレビュー

エンジニアは、実装作業を仕様や保守の観点から総合的にレビューを行う。

* 品質ガイドラインに則った作業が行われているか
    * [プルリクエスト作成](/3.実装/プルリクエスト作成)
    * [コーディングガイド](/3.実装/コーディング)
    * [ソースコード確認](/4.レビュー/ソースコード確認)
* 修正内容の動作確認
    * 仕様期待値の直球確認
    * [関連箇所のチェック](/4.レビュー/関連箇所のチェック)

また、修正箇所の指摘については [修正箇所の指摘](/4.レビュー/修正箇所の指摘) を参照すること。
